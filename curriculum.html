Módulo 0
Fundamentos del Cómputo
Pensamiento Computacional

Cómo piensa una computadora

Una computadora no "entiende" nada. Ejecuta instrucciones sobre datos. Todo lo que ves en pantalla — texto, imágenes, videos — son números representados en binario (0s y 1s). Entender esto cambia cómo piensas sobre el software.

La jerarquía de memoria es la primera cosa que un senior tiene presente: registros → caché → RAM → disco duro → red. Cada nivel es más lento y más barato. Tu trabajo como programador es mover los datos correctos al nivel correcto en el momento correcto.

RAM (Memoria)
Rápida, volátil. Almacena lo que el programa usa ahora. Se borra al apagar. Cara por GB.
Almacenamiento (Disco)
Lento, persistente. Guarda datos entre sesiones. Más barato por GB. SSD > HDD en velocidad.
Caché
Memoria ultra-rápida dentro del procesador. Guarda datos usados recientemente. Mínima.
CPU
Ejecuta instrucciones. Cores = capacidad de paralelismo. GHz = velocidad por ciclo.

Los tres paradigmas algorítmicos

Todo algoritmo — sin importar qué tan complejo sea — se construye con solo tres bloques:

  • Secuencia: Instrucciones ejecutadas en orden, una tras otra.
  • Selección: Una decisión basada en una condición (if/else). El programa toma caminos distintos.
  • Repetición: Un bloque que se ejecuta múltiples veces (bucles). El poder de la computación.
# Secuencia
nombre = "Tony"
saludo = "Hola, " + nombre
print(saludo)  # Hola, Tony

# Selección
edad = 20
if edad >= 18:
    print("Mayor de edad")
else:
    print("Menor de edad")

# Repetición
for i in range(5):
    print(f"Iteración {i}")  # Ejecuta 5 veces

Pseudocódigo: Pensar antes de escribir

Los seniors no empiezan a escribir código inmediatamente. Primero diseñan la solución en pseudocódigo — lenguaje natural estructurado que describe la lógica sin preocuparse por la sintaxis.

# Pseudocódigo para buscar el mayor número en una lista
FUNCIÓN encontrar_mayor(lista):
    SI la lista está vacía:
        RETORNAR error "lista vacía"
    
    mayor = primer elemento de la lista
    
    PARA cada número en la lista:
        SI número > mayor:
            mayor = número
    
    RETORNAR mayor

Un senior ve el código como la última fase, no la primera. Antes de escribir una línea, se pregunta: ¿qué problema estoy resolviendo exactamente? ¿Cuáles son los casos edge? ¿Qué pasa si la entrada está vacía o es inválida? Escribir pseudocódigo primero evita reescribir código real después.

El error más común es confundir RAM con almacenamiento. Si guardas algo en una variable, existe solo mientras el programa corre. Si apagas el ordenador, desaparece. Para que persista, debes escribirlo en un archivo o base de datos.

  • RAM = temporal, rápida. Disco = permanente, lenta. Nunca confundirlas.
  • Todo algoritmo = secuencia + selección + repetición. Nada más.
  • Diseña antes de codificar. El pseudocódigo no tiene sintaxis incorrecta.
  • La computadora ejecuta instrucciones exactamente como se las das. No interpreta intención.

Python Básico
Lenguaje — Python

Todo en Python es un objeto

En Python, las variables no contienen valores — son etiquetas que apuntan a objetos en memoria. Esta distinción parece sutil pero explica comportamientos que confunden a principiantes durante meses.

a = [1, 2, 3]
b = a          # b apunta al MISMO objeto que a
b.append(4)
print(a)       # [1, 2, 3, 4] — a también cambió

# Para copiar realmente:
c = a.copy()   # c es un objeto NUEVO
c.append(5)
print(a)       # [1, 2, 3, 4] — a no cambió

Tipos de datos fundamentales

# Enteros — precisión arbitraria (sin overflow)
edad = 25
year = 2024

# Flotantes — representación aproximada (IEEE 754)
precio = 19.99
print(0.1 + 0.2)  # 0.30000000000000004 — no es un bug, es IEEE 754

# Strings — inmutables, unicode por defecto
nombre = "Tony"
frase = f"Hola, {nombre}. Tienes {edad} años."  # f-strings: siempre usa estos

# Booleanos — True/False (mayúscula)
activo = True

# None — ausencia de valor (no es 0, no es False)
resultado = None  # "aún no hay resultado"

Operaciones que todo programador debe dominar

# String methods más útiles
texto = "  Hola Mundo  "
texto.strip()         # "Hola Mundo" — elimina espacios
texto.lower()         # "  hola mundo  "
texto.split(" ")      # lista de palabras
"Python".startswith("Py")  # True
",".join(["a", "b", "c"])  # "a,b,c"

# Conversiones de tipo
int("42")     # 42
float("3.14") # 3.14
str(100)      # "100"
bool(0)       # False  — 0, "", [], {}, None son falsy
bool("hola")  # True

# Verificar tipos (úsalo con moderación)
isinstance(42, int)      # True
isinstance("hola", str)  # True

Un senior siempre usa f-strings, nunca concatenación con +. Un senior pone type hints en funciones públicas — no porque Python los enforce, sino porque son documentación viva. Y un senior nunca usa nombres como data, info, temp o x en código de producción.

El flotante trampa: Nunca uses flotantes para dinero. 0.1 + 0.2 != 0.3 en casi todos los lenguajes. Usa el módulo decimal de Python para cálculos financieros: from decimal import Decimal; Decimal("0.1") + Decimal("0.2") da 0.3 exacto.

  • Variables son etiquetas, no cajas. Asignar b = a no copia el objeto.
  • Usa f-strings siempre: f"Hola {nombre}", no "Hola " + nombre.
  • Pon type hints en tus funciones desde el principio. Es gratis y documenta el contrato.
  • Nombres descriptivos > comentarios explicando nombres malos.
  • Nunca float para dinero. Usa Decimal o enteros en centavos.
Módulo 2
Control de Flujo
Lenguaje — Python

Condicionales: el arte de las guardias

Los seniors escriben condicionales de forma diferente a los juniors. El patrón más importante es la guard clause (cláusula guarda): retornar pronto para eliminar anidamiento.

# Junior — anidamiento profundo (difícil de leer)
def procesar_pedido(pedido):
    if pedido is not None:
        if pedido["activo"]:
            if pedido["cantidad"] > 0:
                return pedido["precio"] * pedido["cantidad"]

# Senior — guard clauses (fácil de leer)
def procesar_pedido(pedido):
    if pedido is None:
        raise ValueError("Pedido no puede ser None")
    if not pedido["activo"]:
        return 0
    if pedido["cantidad"] <= 0:
        raise ValueError("Cantidad debe ser positiva")
    
    return pedido["precio"] * pedido["cantidad"]

Bucles: más allá del for básico

frutas = ["manzana", "pera", "uva"]

# enumerate — siempre que necesites índice Y valor
for indice, fruta in enumerate(frutas):
    print(f"{indice}: {fruta}")

# zip — iterar dos listas en paralelo
precios = [1.5, 2.0, 3.5]
for fruta, precio in zip(frutas, precios):
    print(f"{fruta}: ${precio}")

# List comprehension — más pythónico y rápido que un for
cuadrados = [x**2 for x in range(10)]
pares = [x for x in range(20) if x % 2 == 0]
mayusculas = [s.upper() for s in frutas]

# Dict comprehension
precios_dict = {fruta: precio for fruta, precio in zip(frutas, precios)}
# {"manzana": 1.5, "pera": 2.0, "uva": 3.5}

Herramientas funcionales: any, all, filter, map

numeros = [2, 4, 6, 7, 8]

# any — ¿alguno cumple la condición?
any(n % 2 != 0 for n in numeros)  # True (7 es impar)

# all — ¿todos cumplen la condición?
all(n > 0 for n in numeros)       # True

# Evitar bucles explícitos cuando no son necesarios
# En vez de:
impares = []
for n in numeros:
    if n % 2 != 0:
        impares.append(n)

# Usar:
impares = [n for n in numeros if n % 2 != 0]  # [7]

# match-case (Python 3.10+) — mejor que cadenas if/elif largas
def describir_codigo(status_code: int) -> str:
    match status_code:
        case 200: return "OK"
        case 404: return "No encontrado"
        case 500: return "Error del servidor"
        case _:   return f"Código desconocido: {status_code}"

La regla del senior: si tu función tiene más de 2 niveles de anidamiento, algo está mal. Usa guard clauses, extrae lógica a subfunciones, o reconsidera el diseño. El código que un senior escribe casi siempre "cabe en la pantalla" sin scroll horizontal.

Nunca modifiques una lista mientras la recorres con for. Python no lanzará error, pero el comportamiento es impredecible. Si necesitas filtrar, crea una nueva lista: activos = [u for u in usuarios if u["activo"]].

  • Guard clauses primero: valida, rechaza, regresa pronto. La lógica feliz al final.
  • Usa enumerate() en lugar de range(len(lista)) siempre.
  • List comprehensions para transformar/filtrar datos. Bucles for para efectos secundarios.
  • Dos niveles de anidamiento máximo. Más que eso: extrae una función.
Módulo 3
Funciones
Lenguaje — Python

El principio de una sola responsabilidad

Una función debe hacer exactamente una cosa. Si no puedes describir qué hace con una oración sin usar "y", la función hace demasiado. Este principio es el origen de casi todo el buen diseño de software.

# MAL — función que hace múltiples cosas
def procesar_usuario(nombre, email, datos_compra):
    if "@" not in email:
        return False
    bd.guardar({"nombre": nombre, "email": email})
    total = sum(item["precio"] for item in datos_compra)
    enviar_email(email, f"Total: {total}")
    return True

# BIEN — cada función hace una cosa
def es_email_valido(email: str) -> bool:
    return "@" in email and "." in email.split("@")[1]

def calcular_total(items: list[dict]) -> float:
    return sum(item["precio"] for item in items)

def crear_usuario(nombre: str, email: str) -> dict:
    if not es_email_valido(email):
        raise ValueError(f"Email inválido: {email}")
    return {"nombre": nombre, "email": email}

Argumentos: la trampa del argumento mutable

# EL BUG MÁS CLÁSICO DE PYTHON
def agregar_item(item, lista=[]):
    lista.append(item)
    return lista

agregar_item("manzana")  # ["manzana"]
agregar_item("pera")     # ["manzana", "pera"] ← BUG

# CORRECTO
def agregar_item(item, lista=None):
    if lista is None:
        lista = []
    lista.append(item)
    return lista

# *args y **kwargs para funciones flexibles
def registrar(*mensajes, nivel="INFO", **metadata):
    for msg in mensajes:
        print(f"[{nivel}] {msg} | {metadata}")

registrar("Inicio", "Conectando", nivel="DEBUG", servicio="auth", version=2)

Funciones puras vs efectos secundarios

# Función PURA — mismo input, siempre mismo output, sin efectos externos
def calcular_precio_con_impuesto(precio: float, tasa: float) -> float:
    return precio * (1 + tasa)

# Función CON EFECTOS SECUNDARIOS — modifica estado externo
contador_global = 0
def incrementar():
    global contador_global
    contador_global += 1

# Los seniors prefieren funciones puras cuando es posible.
# Son predecibles, testeables y reutilizables.

Decoradores: funciones que envuelven funciones

import time
from functools import wraps

def medir_tiempo(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        inicio = time.time()
        resultado = func(*args, **kwargs)
        duracion = time.time() - inicio
        print(f"{func.__name__} tardó {duracion:.3f}s")
        return resultado
    return wrapper

@medir_tiempo
def buscar_en_bd(query: str):
    time.sleep(0.1)
    return {"resultados": []}

buscar_en_bd("SELECT * FROM usuarios")

El test de una buena función: ¿puedes escribir un test unitario para ella sin crear una base de datos, un servidor, o un archivo? Si la respuesta es no, la función está haciendo demasiado o mezclando capas.

Nunca uses global. Si una función necesita modificar estado externo, ese estado debería ser un parámetro o la función debería ser un método de una clase.

  • Una función = una responsabilidad. Si usas "y", divídela.
  • Nunca def f(arg=[]):. Usa None como default para mutables.
  • Prefiere funciones puras. Los efectos secundarios son necesarios pero deben estar controlados.
  • Type hints + docstring = documentación que no se desactualiza.
Módulo 4
Estructuras de Datos
Lenguaje — Python

La pregunta que define la estructura correcta

Antes de elegir una estructura de datos, un senior se pregunta: ¿cuál es la operación más frecuente? Buscar, insertar, acceder por índice, verificar pertenencia.

Lista []
Acceso por índice O(1). Búsqueda O(n). Append O(1) amortizado. Orden importa, duplicados OK.
Diccionario {}
Búsqueda por clave O(1). Inserción O(1). Clave única. Orden preservado (Python 3.7+).
Set {}
Verificar pertenencia O(1). Sin duplicados. Sin orden. Ideal para unicidad.
Tupla ()
Inmutable. Más rápida que lista. Usa para datos que no deben cambiar. Hasheable.
from collections import defaultdict, Counter
from typing import NamedTuple

emails_bloqueados = {"spam@evil.com", "bot@fake.org"}
if "spam@evil.com" in emails_bloqueados:
    print("Bloqueado")

ventas_por_region = defaultdict(list)
ventas_por_region["norte"].append(1500)

palabras = ["python", "es", "genial", "python", "es"]
conteo = Counter(palabras)
conteo.most_common(2)

class Coordenada(NamedTuple):
    latitud: float
    longitud: float
    nombre: str = "Sin nombre"

punto = Coordenada(4.710989, -74.072090, "Bogotá")
print(punto.nombre)

En producción, la estructura de datos incorrecta puede convertir una operación que tarda milisegundos en una que tarda minutos.

  • Verificar si algo existe → usa set o dict, nunca list.
  • Agrupar por clave → defaultdict. Contar → Counter.
  • Datos inmutables con nombres → NamedTuple o dataclass.
Módulo 5
Programación Orientada a Objetos
Paradigmas

OOP no es sobre clases, es sobre responsabilidades

El error de los juniors es pensar que POO = crear clases. La POO real es sobre modelar responsabilidades y relaciones en el dominio del problema.

from dataclasses import dataclass, field
from datetime import datetime

@dataclass
class Producto:
    id: int
    nombre: str
    precio: float
    stock: int = 0
    def tiene_stock(self) -> bool:
        return self.stock > 0
    def aplicar_descuento(self, porcentaje: float) -> float:
        if not 0 <= porcentaje <= 100:
            raise ValueError("Porcentaje inválido")
        return self.precio * (1 - porcentaje / 100)

@dataclass
class CarritoDeCompras:
    usuario_id: int
    items: list = field(default_factory=list)
    creado_en: datetime = field(default_factory=datetime.now)
    def agregar(self, producto: Producto, cantidad: int = 1) -> None:
        if not producto.tiene_stock():
            raise ValueError(f"{producto.nombre} no tiene stock")
        self.items.append({"producto": producto, "cantidad": cantidad})
    @property
    def total(self) -> float:
        return sum(item["producto"].precio * item["cantidad"] for item in self.items)

Herencia vs Composición

class Motor:
    def encender(self) -> str:
        return "Motor encendido"
class Auto:
    def __init__(self):
        self.motor = Motor()
    def arrancar(self) -> str:
        return self.motor.encender()
# Regla: prefiere composición sobre herencia.

La pregunta que hace un senior antes de crear una clase: "¿qué datos van siempre juntos y qué comportamientos los modifican?"

  • Herencia = "ES UN". Composición = "TIENE UN". Prefiere composición.
  • Usa @property para exponer atributos con control.
  • @dataclass para clases que principalmente guardan datos.
Módulo 6
Algoritmos y Complejidad
CS Fundamentals

Big-O: el lenguaje universal del rendimiento

Big-O describe cómo crece el tiempo de ejecución según crece el input. Mide escalabilidad, no segundos.

O(1) — Constante
Acceso a dict, list por índice.
O(log n) — Logarítmico
Búsqueda binaria. 1M elementos → ~20 ops.
O(n) — Lineal
Un loop simple. Búsqueda en lista.
O(n²) — Cuadrático
Loop dentro de loop. Peligroso.
import bisect
usuarios = {"tony": {"edad": 25}}  # O(1)
lista_ordenada = [1, 3, 5, 7, 9, 11, 13, 15]
indice = bisect.bisect_left(lista_ordenada, 7)  # O(log n)

def tiene_duplicados_lento(lista):  # O(n²)
    for i in range(len(lista)):
        for j in range(i + 1, len(lista)):
            if lista[i] == lista[j]:
                return True
    return False

def tiene_duplicados_rapido(lista):  # O(n)
    return len(lista) != len(set(lista))

Cuando un senior ve un bucle dentro de otro bucle en producción, se detiene: "¿puede ser O(n) en lugar de O(n²)?"

  • Bucle dentro de bucle = O(n²). Siempre busca reducirlo con un dict/set.
  • Usa sorted() de Python — Timsort es O(n log n).
  • Búsqueda en lista = O(n). En dict/set = O(1).

Un senior sabe que el esquema de base de datos es el contrato más difícil de cambiar en una aplicación.

El problema N+1

# BUG — N+1 queries
pedidos = db.query("SELECT * FROM pedidos")
for pedido in pedidos:
    cliente = db.query(f"SELECT * FROM clientes WHERE id={pedido.cliente_id}")

# CORRECTO — JOIN
pedidos_con_clientes = db.query("""
    SELECT p.*, c.nombre, c.email
    FROM pedidos p
    JOIN clientes c ON p.cliente_id = c.id
""")

ACID garantiza que tu base de datos no quede en estado corrupto. Las transacciones son obligatorias para operaciones de múltiples pasos.

  • Normaliza el esquema. Datos duplicados crean inconsistencias.
  • Guarda el precio histórico en los ítems de pedido.
  • El N+1 destruye el rendimiento. Siempre usa JOINs.
  • Índices en toda columna usada en WHERE o JOIN.

Calidad

Los tests son la red de seguridad para refactorizar

import pytest

def calcular_precio_final(precio: float, descuento_pct: float, tiene_membresia: bool) -> float:
    if precio <= 0:
        raise ValueError("El precio debe ser positivo")
    if not 0 <= descuento_pct <= 100:
        raise ValueError("Descuento inválido")
    precio_con_descuento = precio * (1 - descuento_pct / 100)
    if tiene_membresia:
        precio_con_descuento *= 0.9
    return round(precio_con_descuento, 2)

class TestCalcularPrecioFinal:
    def test_sin_descuento(self):
        assert calcular_precio_final(100, 0, False) == 100.0
    def test_descuento_50(self):
        assert calcular_precio_final(100, 50, False) == 50.0
    def test_membresia(self):
        assert calcular_precio_final(100, 0, True) == 90.0
    def test_precio_negativo(self):
        with pytest.raises(ValueError, match="debe ser positivo"):
            calcular_precio_final(-10, 0, False)

TDD no es sobre escribir tests antes del código — es sobre diseñar la API antes de implementarla.

  • Estructura AAA: Arrange, Act, Assert.
  • Testea comportamiento, no implementación.
  • Cubre casos edge, no solo el happy path.

# OBSERVER
class EventBus:
    def __init__(self):
        self._listeners: dict[str, list] = {}
    def suscribir(self, evento: str, callback):
        self._listeners.setdefault(evento, []).append(callback)
    def publicar(self, evento: str, datos=None):
        for callback in self._listeners.get(evento, []):
            callback(datos)

# STRATEGY
from abc import ABC, abstractmethod
class EstrategiaDescuento(ABC):
    @abstractmethod
    def calcular(self, precio: float) -> float: ...
class SinDescuento(EstrategiaDescuento):
    def calcular(self, precio): return precio
class DescuentoEstudiante(EstrategiaDescuento):
    def calcular(self, precio): return precio * 0.8

Los patrones son soluciones, no requisitos. No apliques un patrón sin que el problema exista.

  • Observer: múltiples partes reaccionan sin acoplarse.
  • Strategy: variantes de algoritmo intercambiables.
  • YAGNI: "You Aren't Gonna Need It".

En sistemas distribuidos, todo falla

CAP Theorem
Consistencia, Disponibilidad, Tolerancia a particiones. Elige 2 de 3.
Consistencia eventual
Datos consistentes, no inmediatamente. DNS, carritos.

Desacopla productores y consumidores.

Primera regla: no hagas sistemas distribuidos si no lo necesitas. Monolito primero.

  • Idempotencia: N ejecuciones = 1 resultado.
  • Monolito primero, microservicios solo con razón.
